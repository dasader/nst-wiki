# 대시보드 문서 업로드 UI 설계

작성일: 2026-07-06

## 배경

문서 인제스트는 백엔드 `POST /api/v1/ingest`가 이미 처리한다(파일 1개 + `title`
필수·`publisher`·`publish_date`·`tags`·`force`, 해시 중복은 409로 차단). 지금은
curl로만 올릴 수 있어 관리자용 업로드 UI가 없다. 관리 대시보드
(`api/app/static/dashboard.html`, 순수 HTML/JS 단일 파일)에는 태스크 목록·상태
배지·15초 자동새로고침·소스 삭제·원본 다운로드가 이미 있다.

## 목표

대시보드에 문서 업로드 패널을 추가한다. **백엔드·API 변경 없음** — 전부 기존
`POST /api/v1/ingest`로 처리한다.

- 드래그-드롭 또는 클릭으로 여러 파일 선택
- 파일마다 메타데이터(title·publisher·tags·발행일) 개별 입력
- 업로드 전송 진행률(%) 표시, 이후 처리 상태는 기존 태스크 목록이 추적
- 클라이언트 사전검증(확장자·용량), 업로드 전 파일 제거/교체

## 비목표

- 백엔드 변경(용량 제한, 메타데이터 편집 엔드포인트 등)
- 업로드 후 처리 상태(parsing/classifying/staged)를 업로드 패널에 인라인 추적 —
  기존 태스크 목록의 자동새로고침 배지가 담당
- 공개 프론트엔드(:3000)에 업로드 노출 — 업로드는 admin 인증 필요, 대시보드에만 둔다

## UI 구조

기존 "위키 페이지 편집" `<details>` 옆에 "문서 업로드" 섹션을 추가한다.

- **드롭존**: 클릭 시 파일 선택창(`<input type="file" multiple accept=".pdf,.md,.xlsx">`),
  드래그-드롭도 수신. 네이티브 파일 입력 기능 사용.
- **행 목록**: 드롭/선택된 파일마다 한 행. 각 행:
  - `title` 입력(기본값 = 확장자 뗀 파일명, 편집 가능)
  - `publisher` · `tags` · `발행일(publish_date)` 입력
  - **제거** 버튼(드래그 중 제거/교체 = 제거 후 다시 드롭)
  - **상태** 칸: `대기 → 업로드중 nn% → ✓ 큐등록(task 8자리) / ✗ 409 중복 / ✗ 오류(메시지)`
- **배치 상단 "덮어쓰기(force)" 체크박스** 하나: 켜면 모든 전송에 `force=true` 부착.
- **일괄 업로드 버튼**: 유효(사전검증 통과) 행을 순차 전송.

## 클라이언트 사전검증

서버로 보내기 전 브라우저에서 거른다:

- 확장자 화이트리스트 `.pdf`/`.md`/`.xlsx` (서버 `ALLOWED_EXTS`와 동일하게 유지)
- 최대 용량 상수 `MAX_MB`. 서버엔 용량 제한이 없으므로 순수 클라 UX 가드.
  `// ponytail: 개인용 기본값, 서버가 거부하면 상향` 주석. 초깃값 50.

검증 실패 행은 상태 칸에 사유 표시하고 업로드 대상에서 제외한다(silent drop 금지).

## 전송

행마다 `XMLHttpRequest` + `FormData`:

- 필드: `file`, `title`, `publisher`, `publish_date`, `tags`, `force`.
  `source_type`은 서버 기본값(`policy_doc`) 사용 — 폼에 노출 안 함.
- 헤더: `X-Admin-Key`(기존 `#adminkey` 입력값 재사용).
- `xhr.upload.onprogress`로 행의 % 갱신.
- 순차 전송(`// ponytail: 순차, 배치 커지면 병렬화`).
- 성공(200): 응답 `task_id`를 행에 표시, 전송 후 `loadList()` 호출해 새 태스크가
  목록에 뜨게 함. 처리 상태는 기존 배지가 추적.
- 409(중복): 서버 메시지를 행에 표시. 상단 force 체크 후 재업로드로 덮어쓰기.
- 기타 오류: status + 응답 본문(잘라서) 표시.

## 테스트

순수 HTML/JS 단일 파일이라 프레임워크 없이 검증한다. 검증 대상 로직:

- 확장자·용량 사전검증 판정 함수 → 인라인 `assert` 자체검증(`.pdf` 통과, `.txt`
  거부, 초과 용량 거부).

전송·드래그-드롭 같은 DOM/네트워크 상호작용은 수동 확인(대시보드에서 실제 업로드
1건: PDF 큐등록 → 목록에 staged 표시, 중복 재업로드 시 409 → force로 통과).

## 리스크

- 대용량 파일 전송 중 브라우저 메모리: `FormData`가 스트리밍하므로 문제 없음.
- 순차 전송은 파일 많으면 느림 — 개인용 관리 도구라 허용, 필요 시 병렬화(위 ponytail).
